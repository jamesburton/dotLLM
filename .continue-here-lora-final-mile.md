## TL;DR

Phase 4d.5 plumbed the dispatch seam so the pre-quantised activation
buffer (`preQuantNorm`, `preQuantAttn`, `preQuantFfn`, `preQuantSilu`)
flows from each base-projection site into `ApplyLoraDelta`, and added
`LoraDelta.ApplyQ8_0BWithPreQuantX` which routes Q8_0 LoRA stage 1
through `MatMul.GemmQ8_0(preQuantizedInput=xQ8)`. The activation-quant
cost that Agent 7's Phase 4d.4 spike couldn't amortise is now zero.

But the macro-bench regression vs Agent 7's dequant-once path is **still
not closed** — direct kernel probing (M=rank=16, K=hidden=2048,
N=seqLen=512) on Strix Halo shows `MatMul.GemmQ8_0` is ~1.7× slower per
call than the F32 dequant-once GEMM. The Q8_0 integer-dot kernels
(`VecDotQ8_0Avx512_4Rows`, `ProcessAvx512DualBlock`) amortise their
per-block constant cost (Half→F32 scale conversion, Avx2.Sign trick for
signed-byte products, `MultiplyAddAdjacent` dual chain,
`ConvertToVector512Single`, FMA accumulator) across M weight rows. At
M=16 there isn't enough vertical reuse to pay back vs the F32 path's
`TensorPrimitives.Dot` inner loop — and `TensorPrimitives.Dot` is itself
heavily tuned by .NET 10 RyuJIT.

So Gap 2's win is gated behind `DOTLLM_LORA_FORCE_Q8_PREQUANT=1` (off by
default) until the underlying Q8_0 stage-1 kernel is rewritten for
tiny-M geometry.

## What landed at Phase 4d.5

- `src/DotLLM.Cpu/Kernels/LoraDelta.cs` — new
  `ApplyQ8_0BWithPreQuantX(byte* xQ8, byte* bWeight, void* aWeight, …)`
  overload. Stage 1 = `MatMul.GemmQ8_0(weightsQ8=bWeight, …,
  preQuantizedInput=xQ8)`. Stage 2 unchanged (same `Stage2` helper as
  Phase 4d.4). Single-threaded by design — pool dispatch overhead
  (~100 us round-trip) is too high for M=rank=16 work-per-thread on

- `src/DotLLM.Models/Architectures/TransformerModel.cs` — `ApplyLoraDelta`

  q/k/v/o + gate/up/down call sites pass their already-quantised

## Why the kernel-level probe matters

Macro-bench numbers on this fixture (Llama-3.2-1B-Q8_0, 512-token
prefill) carry ±10 tok/s of variance run-to-run on the same code path —
high enough that single-run comparisons across agents are unreliable.
Agent 7's reported 123.89 tok/s for LoraQ8_0 prefill vs our 113.56
(med) / 115.29 (best) is within the noise floor.

The deterministic signal is the kernel probe (microbench, single
shape, hot cache). It pins down that the underlying `MatMul.GemmQ8_0`
is the limiter — not the dispatch refactor, not the host-dequant
path, not anything else Phase 4d.5 changed.

The probe used:
```
M=rank=16, K=hidden=2048, N=seqLen=512
80 iterations, 5 warmup
Strix Halo (Ryzen AI Max+ 395, AVX-512F+CD+BW+DQ+VL+VBMI)
.NET 10.0.3 RyuJIT
```
Result (medians): dequant-once F32 = 0.47 ms/call,
Q8_0 GEMM with pre-quant = 0.97 ms/call. The Q8_0 GEMM should be
~3-4× faster on byte volume alone (weights are 4× smaller, activation
is 4× smaller); its 2× slowdown vs F32 tells us the kernel is
constant-cost-bound for tiny M.

(The probe was prototyped at `benchmarks/LoraQ8Stage1Probe/` but
deleted before commit — it's a single-shape probe, not a long-lived
bench. Re-create on demand.)

## Proposed next step — tiny-M Q8_0 stage-1 kernel

The existing `MatMul.GemmQ8_0` (specifically `ComputeGemmTiled` and
`ComputeRows`) is **row-tile-then-tokens**: outer loop iterates over
M-row tiles, inner loop iterates over N input tokens. For each
(tile, token) pair it computes all `tileRows` outputs at once via
`VecDotQ8_0Avx512_4Rows`, which loads x once per block-pair and
processes 4 weight rows in parallel.

When M=tileRows is small (e.g. 16), each per-(tile,token) call does:
- 4 batches of `VecDotQ8_0Avx512_4Rows`
- Each batch: 32 dual-block iterations (K/64) × (2 x-loads + 8 weight
  loads + 8 sign ops + 8 MAA + 4 ConvertToSingle + 4 FMA)
- Total per token: ~256 vec256 loads + ~600 ops

The work-per-token is modest, but each token call re-reads the weights
(34 KB) and the x (2.2 KB). At seqLen=512 the cumulative weight reads
are 512 × 34 KB = 17 MB — well beyond L2.

For the F32 dequant-once path, the same per-token work is fewer ops
(~16 dot products × 2048 floats each), but `TensorPrimitives.Dot`
batches them through AVX-512 32-float-wide accumulators with no
per-block constant. Net win for F32.

### Sketch — `GemvQ8_0PerToken` for LoRA stage 1

A kernel tuned for tiny M = rank, large N = seqLen:

```
for each token t in [0, seqLen):
    xQ8_t = xQ8 + t * q8RowBytes        // 2.2 KB pre-quantised activation
    out_t = tmp + t * rank              // [rank] F32 destination

    for each weight row r in [0, rank):
        // GEMV inner: dot(B[r,:], xQ8_t) returns F32
        // Reuse VecDotQ8_0Avx512 (single-row variant) — already exists
        // in MatMul.cs at line 149.
        out_t[r] = VecDotQ8_0Avx512(B_q8 + r * rowBytes, xQ8_t, blockCount)

    // No 4-row batching → no per-quad acc setup cost
    // Each row's dot is a clean linear sweep of blockCount blocks
```

Why this might be faster:
- Per-token activation `xQ8_t` is read ONCE per token and reused across
  all `rank` weight rows naturally (vs `VecDotQ8_0Avx512_4Rows` which
  reads x once per block-pair × 4 rows but at the cost of more vector
  registers held live).
- For M < 32 the 4-row batching's vertical reuse is wasted — only 4
  groups of 4 rows means only 4 outer batches per token.
- The single-row `VecDotQ8_0Avx512` is simpler — fewer dependent chains,
  potentially better OOO scheduling on Zen 5.

### Acceptance criteria for the new kernel

- Kernel-level probe (same M=16, K=2048, N=512 shape): at least
  parity with `MatMul.GemmF32` dequant-once path (≤ 0.47 ms median),
  ideally 0.5× (the byte-bandwidth-fair win).
- Macro-bench rerun: Q8_0 LoRA prefill regression ≤ −10% vs NoLora on
  the standard Llama-3.2-1B-Q8_0 fixture.
- Bit-parity: `LoraDeltaQuantizedQ8_0Tests` still passes (the new path
  must match Q8_0 round-trip semantics same as the existing one).

### Alternative — dequant B once into Q8_0 layout (not F32)

Another angle: the dequant-once path reads B's 34 KB Q8_0, expands to
128 KB F32 scratch. What if we instead stay in Q8_0 land but lay B out
**interleaved by rank** so the per-token integer-dot processes
`rank` consecutive bytes (per K-block) in a single AVX-512 instruction?
This is what dotLLM already does for the base Q8_0 GEMM via the
`R4` (4-row repack) layout — see `MatMul.GemmQ8_0R4Interleaved`. The
LoRA adapter could store B in `R8` or `R16` repacked layout if rank
permits (rank=16 is a natural fit).

This shares the spirit of llama.cpp's `mul_mat_vec_q` family — wide-M
GEMV kernels that consume Q8_0 weights in cache-line-friendly
interleaved layouts. Costs:
- Adapter-load complexity: must repack B from `[rank, K]` row-major
  Q8_0 into `[K/32, rank, 32]` block-interleaved Q8_0 (one-shot per
  adapter, negligible).
- New stage-1 kernel signature; existing `MatMul.GemmQ8_0R4Interleaved`
  is hard-coded for R4 (4-row groups).

Either approach (tiny-M GEMV-per-token OR R16-interleaved B layout) is
a self-contained kernel-side spike, separable from any other LoRA work.

## Anti-goals (preserved from Phase 4d.4)

- Don't change `LoraWeightDType` storage layouts — Q8_0 is `[rank, K]`
  row-major, 34 bytes per 32-element block, matching GGUF Q8_0. Any
  new interleaved layout MUST be added as a new dtype variant
  (`Q8_0_R16` or similar), not by silently repacking on load.
- Don't ship a regression vs Agent 7's Phase 4d.4 numbers as the
  default path. Gate any new kernel under an env var until kernel-level
  probe + macro-bench both validate it.
- Don't touch the Vulkan path further on this work item — Phase 4d.5
  Gap 1 closed it (host-dequant on upload + F32 fused-delta shader).
- Don't add F16/BF16-LoRA-specific fast paths at the dispatch seam —
  those dtypes aren't memory-bandwidth-bound at rank=16 (B fits in L1).
