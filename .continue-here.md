- **`745b091`** — HF safetensors Llama-family RoPE: `Norm` → `NeoX`. HF uses
  `rotate_half` (halves); GGUF converter permutes Q/K weights so GGUF stays
  on `Norm`. TinyLlama 3/5 → 5/5 argmax; 30× tighter drift. Also fixes
  Mistral/Mixtral/DeepSeek-V2 (latent, no real-weight tests for those yet).
- **`576ea91`** — DeepSeek shared-expert loader recognises fused-plural
  naming (`mlp.shared_experts.{gate,up,down}_proj`, no numeric index) per
  HF's `DeepseekV2MoE.__init__`. Shared-expert branch was silently no-op'ing
  on all 26 MoE layers. 1/5 → 5/5 argmax; 91× tighter drift.

## Open items (pick any; all independent)

### 1. ~~CPU decode perf~~ — closed by `173fd29`

The 4x gap was worker-pool coordination, not kernel quality. PerfView
kernel sampling showed 76% of dotLLM CPU in the SpinWait spin-loop;
`ThreadPoolDispatchBenchmarks` isolated the cause (cache-line
contention on `_dispatchGeneration` at 32 threads); one-line default
change (`_decodeThreadCount` cap at 8 when no `NumaTopology`) closed
the gap to within 3% on cold runs.

Remaining cheap wins *if* we regress or see a similar pattern on a
different host:

- **1b. Size-gated single-threaded fast path.** Skip `Dispatch` entirely
  when per-thread work < ~500 ns (MatMul call sites already know M×K).
  Microbench showed dispatch cost exceeds single-thread work from 4T up,
  so a small-matmul fast path is almost always a small win. Not
  currently needed — we're at parity.
- **1c. Cache-line-pad hot fields in `ComputeThreadPool`.** Pad
  `_dispatchGeneration`, `_activeWorkerCount`, `_dispatchThreadCount`
  into separate cache lines. Likely moves the 32T SpinWait curve back
  into a usable shape if someone opts in via `DecodeThreadCount`
  override. Still not needed at default settings.
- **1d. AVX-512 R4 kernel.** The R4-layout path only has
  `VecDotQ8_0Avx2_4RowsR4`, no AVX-512. PerfView showed the non-R4
  AVX-512 path dominates actual kernel time (827 vs 122 samples) for
  SmolLM-135M on this CPU, so this is a small absolute win. ~1-2 days.

### 2. CPU prefill perf gap — still open

The llama-bench numbers before this session showed ~7x prefill gap
(1465 vs 205 tok/s). The decode-only fix in `173fd29` doesn't touch
prefill (prefill uses EventBased mode which was never on the collapse
curve). Re-measure `llama-bench` on HEAD first to confirm the gap is
still there. If so, tiled GEMM with 8×8 or 16×16 cache blocking in
`MatMul.cs` is the next target. 1.5–2x expected.

### 3. Vulkan LLM kernels (biggest long-term win on this box)

order documented in `CONTINUE_ON_CUDA.md` §Reciprocal. ~3-4 weeks for a
complete Llama-family Vulkan forward. CUDA kernels in
`native/kernels/*.cu` transpile mechanically to GLSL/SPIR-V.

### 9. Perf profile (ALWAYS do before touching perf code)

`dotnet-trace` **with managed-stack sampling is insufficient** for this
codebase — the SIMD hot path is JIT-emitted intrinsics that the sampler
cannot walk, so 100% of leaf self-time shows as `UNMANAGED_CODE_TIME`.
Inclusive attribution to managed callers still works (see
`.perf-runs/cross-lib-20260424/profile-summary.txt`) but leaf-level
cycles-per-element needs **PerfView** with kernel-mode CPU sampling:

```
# Once: install PerfView (https://github.com/microsoft/perfview/releases)
PerfView.exe /ThreadTime /DumpHeap=false /NoGui collect \
    .perf-runs/<tag>/dotllm-kernel.etl

# While collecting, send a chat request that will run for ~20 s
curl -X POST http://127.0.0.1:8000/v1/chat/completions \
    -H "Content-Type: application/json" \
    -d '{"model":"SmolLM-135M","messages":[{"role":"user","content":"..."}],"temperature":0,"max_tokens":192}'
```

In PerfView: open the ETL → "CPU Stacks" → IncPats = `dotnet`. Top leaf
frames should be `VecDotQ8_0Avx2_4RowsR4`, `VecDotQ8_0Avx512_4Rows`,
`QuantizeF32ToQ8_0Avx512`, attention tile loops. Use "Fold %" 1 or lower
to avoid hiding small helpers.
