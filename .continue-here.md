### MMQ Q4_K dp4a kernel (`6368950`) — graph 188 → 200 tok/s

`native/kernels/quantized_gemv_mmq.cu`. Input activation quantized to INT8
per 32-element chunk (matches Q4_K sub-block layout); per-row dot product
via `__dp4a` (packed 4×INT8 multiply-add, sm_61+) instead of FP fmuladd.
Q4_K's per-sub-block min folded as `dmin * m_s * Σxq * dx_c` — single fmaduladd
per sub-block. Tile parallelism: 4 output rows per CUDA block, 256 threads
shared across them via shared-memory accumulators (avoids the "254/256 threads
idle" failure mode at SmolLM's 2-superblock-per-row regime).

Best per-category win: **MlpDown −21 % (1.901 → 1.501 ms)** because its
inputDim=1536 = 6 super-blocks/row (vs 2 for the projections) — more work
per row → MMQ amortization kicks in. QkvProj/MlpUp/OProj see smaller wins
(−3 to −7 %). LmHead unaffected (Q6_K weight, MMQ Q4_K-only for now).

`HasMmqQ4K` accessor + `LaunchQuantizedGemvMmq` launcher; `Project()` and
the fused QKV/GateUp paths route Q4_K decode through MMQ when available.
`DOTLLM_DISABLE_MMQ_Q4K=1` env var forces legacy kernel for A/B comparison.

Logits stay bit-identical to legacy kernel within `Tight` tolerance — input
INT8 requantization noise is dominated by existing Q4_K_M weight-quant noise
budget.

### Norm kernel rewrite (`facbcd7`) — kernel-level cleanup

`rmsnorm_f16` / `fused_add_rmsnorm_f16` / `rmsnorm_f32` / `rmsnorm_f32in_f16out`
/ `per_head_rmsnorm_f16` rewritten with vectorized `half2`/`float2` loads
+ `__shfl_xor_sync` symmetric warp reduction + folded-`1/n` rsqrt arg.
`fused_add_rmsnorm_f16` extra trick: caches FP32 sum in dynamic shared
memory between pass 1 (residual + x) and pass 2 (scale by rms_inv). Pass 2
reads un-truncated FP32 sum from shmem instead of re-reading freshly-stored
FP16 residual — saves one global read + FP16↔FP32 conversion per element.

Per-call kernel speedup is real but small (~1-3 µs each). End-to-end
contribution to graph tok/s is below noise floor at SmolLM scale; will
matter more on larger hidden sizes (Llama-7B+). Adds 4 new kernel
equivalence tests (`RmsNormF16_*`, `FusedAddRmsNormF16_*`).

### Pre-Q8_1 input quant + MMVQ-large default ON (`7c47493 + 1ebae6f + fc79371`) — Qwen3-8B 11.5 → 30.9 tok/s (2.69×)

**The breakthrough commit of the session.** Implements research note §H4
(`docs/perf/MLPUP_GEMV_GAP.md`): a single `quantize_x_to_q8_1` kernel
quantizes the activation `x[k]` to INT8 + per-32-element FP16 scale +
per-half-chunk sums **once per fused-GEMV-input** (not once per CUDA block).
Then `_preq` variants of all MMQ + MMVQ-large kernels skip Stage 1 and
read from the precomputed scratch.

This unblocks the MMVQ-large path that previously regressed 2× without
the prerequisite: MMVQ-large is 1-row-per-block, so without pre-quant the
Stage 1 input-quantization runs `n` times per GEMV (vs `n/4` for MMQ-4-rows).
Pre-quant makes it 1 time, no matter the kernel topology.

**Wiring:**
- `CudaForwardState.PreQ8_1Scratch` — single scratch buffer sized for
  `max(hidden, intermediate)` elements + per-chunk metadata. ~14.6 KB on
  Qwen3-8B.
- `LaunchQuantizeXToQ8_1` runs in `MaybePreQuantize` helper before each
  fused QKV/GateUp/Down/O/LmHead GEMV. ~145 extra launches per forward
  on Qwen3-8B (4/layer × 36 + 1 LmHead) = ~3.2 ms WDDM overhead, dwarfed
  by the GEMV savings.
- `LaunchQuantizedGemvMmq` overload accepts the scratch pointer; routes
  to `_preq` kernel variant. **MMVQ-large is now default ON for k≥1024**;
  the previous `DOTLLM_ENABLE_MMVQ_LARGE_*` env vars are replaced by
  `DOTLLM_DISABLE_PREQ8_1` (forces on-the-fly path) and the existing
  `DOTLLM_DISABLE_MMVQ_LARGE_*` (forces MMQ-4-rows even with pre-quant).
- Pre-quant is gated on `inputDim ≥ 1024` to keep SmolLM (k=576) on the
  unchanged code path — no perf delta at small scale.

|---|---|---|---|

### MMQ_MAX_CHUNKS bug fix (in `f9b8294`) — silent correctness fix on k≥8192 GEMVs

The `MMQ_MAX_CHUNKS=256` constant in `quantized_gemv_mmq.cu` capped the
per-block shared-memory `s_xq[256*32]` buffer at k=8192 (256 chunks of 32
elements). On Qwen3-8B, `MlpDown` is `[hidden=4096 × intermediate=12288]`
GEMV — k=12288 → **384 chunks**. The kernel's Stage 1 input-quantization
loop wrote past the `s_xq` buffer (chunks 256..383 trashed adjacent shmem
or simply weren't stored), and Stage 2 read uninitialized values for
those super-blocks. Output was visibly fast (no segfault — shmem
overruns are silent) but mathematically wrong: roughly 1/3 of MlpDown
contributions came from garbage. Generated tokens looked plausible-shaped
but the per-token logits were skewed.

The pre-fix Qwen3-8B "baseline" of 10.5 tok/s was on this garbage. The
corrected baseline (MMQ_MAX_CHUNKS bumped to 384) is **11.5 tok/s** with
correct logits. SmolLM-135M (k=576, 18 chunks) was unaffected.

This is a real regression risk for anyone targeting larger-k models on
this branch's history before commit `f9b8294`. The fix is permanent;
we'll bump again to support k≥12288 with margin if Llama-70B-class arrives.

### MMVQ-large opt-in kernels (in `f9b8294`) — structurally correct, regress today

Implements the 1-row-per-block, 128-thread MMVQ kernel for Q4_K/Q5_K/Q6_K
recommended by the research note (`docs/perf/MLPUP_GEMV_GAP.md`). On
Qwen3-8B with MMVQ-large enabled (`DOTLLM_ENABLE_MMVQ_LARGE_*=1`):
**MlpUp 79 ms/token vs default 35 ms — a 2× regression.**

Why: dotLLM does Stage 1 (input quantization to INT8) **on-the-fly inside
every block**. Going from 4-rows-per-block to 1-row-per-block multiplies
Stage 1 work 4× (the input is the same; we re-quantize it 4× across more
blocks). llama.cpp avoids this with a separate pre-Q8_1 quantization
kernel that runs once per forward and is shared across all GEMVs in the
layer (their §H4 in the research note, deferred). Without that
prerequisite, the structural change alone regresses.

**Decision: kernels stay in tree, gated by opt-in `DOTLLM_ENABLE_MMVQ_LARGE_<QT>=1`
(default OFF).** Disable knobs `DOTLLM_DISABLE_MMVQ_LARGE_<QT>=1` take
precedence for forced A/B testing. When pre-quantization (§H4) lands,
flipping the default ON should unlock the 2-4× win the research projected.

9 new equivalence tests in `CudaMmqKernelTests` cover Q4_K/Q5_K/Q6_K at
Qwen3-8B-class shapes (n∈{4096, 11008, 24576}, k=4096) — all pass with
0.5-1% peak-relative drift, same noise budget as the existing MMQ tests.

### Per-row K-quant embedding lookup (`e9f28c0` + `2446ae7`) — saves 1.16 GiB on Qwen3-8B

Adds CUDA kernels `embedding_lookup_q4_k_f16`, `_q5_k_f16`, `_q6_k_f16`
that match llama.cpp's `get_rows_q*_K` pattern: one block per token,
256 threads each emitting one element of each super-block in the row.
Per-superblock arithmetic is bit-identical to the existing
`dequant_q{4,5,6}_k_f16` kernels (3 new equivalence tests prove it).

Wired into `CudaWeights.LoadFromGguf`: when token-embed quant type is
Q4_K/Q5_K/Q6_K and `hidden % 256 == 0`, **skip the load-time FP16 dequant
of the entire vocab×hidden table** and use the per-row lookup at forward
time instead. Falls back to bulk dequant when hidden isn't a 256
multiple, or when `DOTLLM_DISABLE_EMBED_ROWLOOKUP=1`.

VRAM impact (measured on Qwen3-8B Q4_K_M, vocab=151936, hidden=4096):
- Bulk-dequant (`DOTLLM_DISABLE_EMBED_ROWLOOKUP=1`): 7266 MiB resident
- Per-row lookup: **6078 MiB resident → 1188 MiB / 1.16 GiB saved**

End-to-end perf cost: <1 µs/token (the Embed category was 0.7% of GPU
time, stays at ~0.6% post-change). Standard practice in inference
engines (llama.cpp does this for all K-quants).

### Weight dedup vs packed buffer (`15becbf`) — saves ~2.5 GB on 8B-class models

Replaced the "7 separate quantized uploads + DtoD-pack" pattern in
`CudaWeights.LoadFromGguf` with a try-pack-first approach: when QKV (or
Gate/Up) share quant type + input dim and a GEMV kernel exists, allocate
ONE packed buffer and upload Q/K/V directly into it via H2D copies at row
offsets. The per-tensor `qQuant`/`kQuant`/`vQuant` device pointers become
slices into the packed buffer — bit-identical layout, zero data copy. Only
the packed allocation is in the lifetime tracker. Falls back to per-tensor
uploads when packing isn't possible. O and Down stay per-tensor (no pack
to dedup against).

Two diagnostic helpers added: `AllocOrThrowWithContext` and
`MemcpyHtoDOrThrowWithContext` so OOM failures report the requested size
+ free/total VRAM at the failure moment.

Resident-VRAM impact:
- SmolLM-135M Q4_K_M: ~30 MB saved (negligible at this scale).
- Qwen3-4B Q4_K_M: ~1.4 GB saved.
- Qwen3-8B Q4_K_M: **~2.5 GB saved** (was OOMing at ~2.5 GiB resident
  during weight upload; now lands at ~5.3 GiB resident and runs).

End-to-end perf: SmolLM-135M Q4_K_M unchanged (eager 117 → 111 within
noise, graph 217 unchanged). Bit-identical logits — 47/47 GPU tests pass.

### MMQ Q5_K + Q6_K (`d0004c2`) — graph 206 → 217 tok/s (+5.4%)

Extends the dp4a MMQ pattern from Q4_K to Q5_K and Q6_K in
`native/kernels/quantized_gemv_mmq.cu`. Q5_K reuses the same 32-element
chunk granularity as Q4_K (sub-blocks are also 32 elements); weight bytes
packed in input memory order [lo_e, hi_e, lo_o, hi_o] to match the per-chunk
INT8-quantized x stream. Q6_K sub-blocks are 16 elements so each 32-element
input chunk holds two sub-blocks with separate scales — Stage 1 input-quant
pass tracks per-half-chunk sums (sx_a, sx_b) by stopping the warp reduction
at offset=8 (lanes 0..15 hold one half, lanes 16..31 the other). Q6_K has
no dmin term — math simpler than Q4_K/Q5_K.

Wiring: `HasMmqQ5K` / `HasMmqQ6K` accessors + `DOTLLM_DISABLE_MMQ_Q5K` /
`Q6K` env vars; new `HasMmq(QuantizationType)` helper used by `Project()`
and the fused QKV / Gate-Up paths to route any MMQ-supported type through
the dp4a kernel. `CudaModule.TryGetFunction` so a stale PTX (Q4_K-only MMQ)
loads gracefully rather than crashing startup.

End-to-end win on SmolLM-135M Q4_K_M: graph 206 → 217 tok/s. **LmHead is
Q6_K** in this model (`output.weight` block) so MMQ Q6_K activates and
produces the bulk of the 5.4% lift. Eager (clean) within noise (117.9 →
116.7) — the existing FP-fmuladd path was already cuBLAS-fast for Q6_K at
n=49152 row-parallel scale.

Validation: 6 new `CudaMmqKernelTests` (Q5_K + Q6_K × {(4,256),(8,512),
(64,1024)}) — synthetic random-byte weights vs the legacy `quantized_gemv`
kernels; peak-relative drift 0.4–1.9% across all cases (consistent with the
INT8 input requantization noise budget, same as Q4_K MMQ). End-to-end logits
parity on SmolLM-135M Q4_K_M still passes within `Tight` tolerance.

### KvUpdate fusion (`af8b2b4 + b354700`) — eager-path +27% over post-graph baseline

Single fused decode-step kernel `fused_rope_kv_write_f16` (and `_dyn` variant
for graph) replaces 3 eager launches per layer (`rope_f16` + 2× cuMemcpyDtoDAsync
into KvCache) with one. Four thread regions in one grid: Q rotation pairs (in
place on `qSrc`), K rotation pairs (rotated value written **directly** to
cache row `kCacheBase + cachePos*kvStride`), K tail copy (no-op when
`ropeDim == headDim`, common), V plain copy (`vSrc → vCacheBase + cachePos*kvStride`).
576 threads = 3 blocks for SmolLM-135M.

Eager (clean) jumped 96 → 118 tok/s (+23%). KvUpdate category collapsed from
**16.1% to 3.9%** of GPU time (2.58 → 0.54 ms/token). Graph path unchanged
(already amortised launch overhead).

Decode-only (gated on `seqLen == 1`) and FP16 cache only (`kvCache is CudaKvCache`,
not the quantized subclass — that one has its own graph-friendly path now).
Prefill keeps the bulk-memcpy fast path.

### Quantized KV cache → graph (`af22540 + 7edb993`) — 3.09× for KV-quant configs

Move `CudaQuantizedKvCache` per-layer state from host (`_layerQuantizedLength[]`,
ring index) to device-resident counters that share the existing `_decodePosDevice`.
New `_dyn` kernel variants in `quant_kv.cu` and `kv_write.cu`:
- `quant_f16_to_q8_0_dyn` / `_q4_0_dyn` — read decode pos from device pointer,
  predicate "evict?" device-side (window not full → no-op early return)
- `kv_write_one_f16_ring` — writes new K/V to ring slot `pos % windowSize`
- `kv_dequant_q8_0_dyn` / `_q4_0_dyn` — dequant `[0, max(0, pos+1−windowSize))` prefix
- `kv_window_to_scratch_dyn` — scatter live FP16 ring rows to scratch positions

Critical ordering invariant in `UpdateDeviceForGraph`: **quantize-on-evict
must run BEFORE ring-write** — otherwise the about-to-be-evicted slot gets
overwritten by the new row before being captured into the Q-cache. Synthetic
test (`UpdateDeviceForGraph_MatchesEager_AcrossEvictionBoundary`) catches this
within the first eviction cycle.

Q8_0 KV + graph: **191.0 tok/s vs 61.8 tok/s eager = 3.09×.** Matches FP16+graph
within noise (5.24 vs 5.28 ms / token), confirming kernel dispatch is no longer
the bottleneck for KV-quant configs either.

### CUDA Graphs decode (the big win, 2.16×)

`c591daf + dd04797 + d941329`. Decode forward is launch-overhead bound on Windows WDDM
(~22 µs/launch × ~400 launches/decode ≈ 9 ms). `cuStreamBeginCapture` → eager forward
→ `cuStreamEndCapture` → `cuGraphInstantiateWithFlags` runs once on the first decode
step after prefill; subsequent steps `cuGraphLaunch` a single stream packet.

**Key trick (option 3 from the design doc):** seq_kv and the RoPE/attention
position offset are **device-resident** (two 4-byte device ints, `_decodePosDevice`
and `_decodeSeqKvDevice`). Host bumps both via `cuMemcpyHtoD_v2` (~1 µs each)
before each replay. Kernel arg pointers bake into the graph at instantiate time;
the values they dereference do not. Topology stays invariant across decode steps.

`attention.cu` body extracted into a `__device__ __forceinline__` helper; two
`extern "C"` entry points: `attention_f16` (scalar seq_kv, prefill / eager decode)
and `attention_f16_dyn` (reads `seq_kv_ptr`, `position_offset_ptr`). New
`native/kernels/kv_write.cu` (`kv_write_one_f16`) replaces the eager
`cuMemcpyDtoDAsync` whose host-computed dst would clobber the same row each
replay — kernel computes `dst = base + posPtr[0] * stride` device-side. PTX
rebuilt for `compute_75` via `build_ptx.bat`.

`UseGraphCapture` flag on `CudaTransformerModel` enables it. Only the standard
`CudaKvCache` path is graph-capable today (see Open §1).

### Fused QKV + Gate/Up GEMV (`c5fedf8 + ce83ce5`)

Eager decode path packs Q/K/V quantized weights into one `[(n_q + 2*n_kv) × hidden]`
matrix at load (`CudaWeights.LoadFromGguf` → `TryPackThree`/`TryPackTwo`,
byte-concat along N via `cuMemcpyDtoDAsync_v2`). Single quantized GEMV per
group; downstream consumers (RoPE, KV update, attention) take pointer slices
into a packed scratch on `CudaForwardState`. Same trick for gate+up.

Works because every GGUF quantized format independently quantizes each row
along K — concatenating along N just stacks rows. Existing
`quantized_gemv_*` PTX kernels work unchanged with bigger N. Memory cost on
SmolLM-135M Q4_K_M: ~9.3 MB Qkv + ~30 MB GateUp = trivial.

**Decode-only.** Prefill (seqLen > 1) keeps per-tensor cuBLAS HGEMM — packing
along N would make rows non-contiguous per-token. Mixed-quant configs cleanly
fall through to 3/2-call path via null-pointer gate.

### Per-category decode profiler (`130ea21`)

`profile-cuda-decode` subcommand on benchmarks runner. Two layers of timing:
total GPU wallclock via cuEvent bracket, plus 12-bucket per-category breakdown
via `MarkProfile(ProfileCategory)` between kernel groups in `Forward`. Walks
event pool after `cuStreamSynchronize` to compute medians. Subcommand modes:
`--graph`, `--compare` (eager + eager-clean + graph side-by-side),
`--no-profiling` (true wall without cuEventRecord overhead). `CudaInferenceBenchmarks`
gets BenchmarkDotNet coverage.

### CUDA 13 cuBLAS resolver fix (`55b2a65`)

CUDA 13 moved cublas64_13.dll, cudart64_13.dll, nvrtc into `bin\x64\` on Windows
and the installer no longer adds that subdir to PATH. `CudaLibraryResolver`
falls back to probing `%CUDA_PATH%`, `%CUDA_HOME%`, and any `CUDA_PATH_V*`
under both new (`bin\x64`) and legacy (`bin`) layouts. Verified on a clean
PATH with no CUDA entries.

**Graph capture re-evaluated:** at the new speeds, graph is now **never

- SmolLM (small k, lots of launches): graph **2.07×** — biggest win, as
  expected (launch-overhead-bound regime).

per-model). Could revisit a default-ON behavior in a follow-up: graph

**Where dotLLM lands** vs llama.cpp on RTX 3060 (third-party numbers
for context, not run on this box):
- SmolLM-class: not commonly benched at this scale — 225 tok/s is fast.
- 4B-class: llama.cpp typically reports ~50-70 tok/s — dotLLM 54 graph
  is in range.
- 8B-class: llama.cpp typically reports ~25-42 tok/s — dotLLM 33 graph
  is **inside** the range, modestly mid-pack.

The 8B Q4_K_M path is now competitive — we're inside the noise band of
the leading reference impl, on the same hardware.

**The graph→eager crossover is dramatic.** At k=576, decode is launch-overhead
bound (~9 ms / 13 ms of decode wall is launch dispatch) and graphs collapse it
to ~5 ms. At k=2560+ the launches are buried in compute and graph capture's
fixed instantiate cost actually slightly hurts. So `UseGraphCapture` should
remain a per-model knob; the engine should not assume graph is always wins.

(~25–40 tok/s for Qwen3-8B Q4_K_M typical), dotLLM is leaving roughly
