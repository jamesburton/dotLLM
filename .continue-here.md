## Session arc (01d03a4 → d7accc7, ~24 commits)

### CPU decode: 8.66 → 33.76 tok/s (~4× speedup; within 3% of llama.cpp)

Cross-library OpenAI harness, SmolLM-135M Q8_0, 192 output tokens, cold run:

| Backend | toks/s | wall |
|---|---:|---:|
| llama.cpp CPU (`-ngl 0`) | 34.73 | 5.53 s |
| dotLLM CPU (after `173fd29`) | **33.76** | 5.69 s (cold), 4.84–4.92 s (warm prefix) |
| dotLLM CPU (before `173fd29`) | 8.66 | 22.17 s |

**Root cause:** `ComputeThreadPool._decodeThreadCount` fell back to `threadCount` (32) when no `NumaTopology` provided. PerfView kernel sampling showed 76% of dotLLM CPU in the `SpinWait` spin-loop — 31 threads hammering one cache line on `_dispatchGeneration` + contended `_completion.Signal()`. `ThreadPoolDispatchBenchmarks` (commit `e9c93d2`) quantified per-dispatch cost at 333 µs on 32T SpinWait vs 1 µs at 8T. One-line fix in `173fd29`: cap the default to `min(8, threadCount)`, matching the heuristic already used when a topology IS provided (memory channels × 2).

### CPU prefill: 334 → 898 tok/s (2.7×; gap 6.6× → 2.4×)

`PrefillBenchmarks` (pp512-equivalent, commit `ad507f0`). Root cause was NOT the expected cache blocking — weight tiles already fit in Zen 5's L2. Real cause: `GemmTiledQ8Worker` partitioned only by tile, so for narrow-M projections (QKV/O at 192–576 rows with `ComputeTileM=256`), `totalTiles` ∈ {1, 3} and 29 of 32 threads idled on the barrier. Commit `f9b1ff6` splits work across both M (rows) and N (tokens) when `totalTiles < threadCount`. Per-projection GFLOPS: Q/K/V 3.9–5.6×, Gate/Up 2.0–2.3×, Down 3.2×, lm_head 1.4× (wide M, already OK).

### Prefix-cache correctness fix (unblocked honest benchmarking)

Commit `53cee5d` — `ResolveKvCache` was accepting any cached entry whose `MaxLength ≥ promptLen`, with zero room for generation. After the server's warmup request primed the cache at ~34 slots, every subsequent chat completion silently truncated at ~12 tokens regardless of `max_tokens`. Fix clamps to `MaxLength ≥ promptLen + maxTokens`; regression test in `PagedKvCacheIntegrationTests`. Introduced in 17bc4383 (Step 36: Paged KV-cache). Only affected `/v1/chat/completions` because the harness's warmup chat primed the cache; `/v1/completions` was unaffected because no matching prefix.

### Full F32 Vulkan forward (10 kernels, end-to-end decode parity on SmolLM-135M)

Before this session: 1 kernel (`Add`). After: 10 kernels + a validated `VulkanTransformerModel` that decodes SmolLM-135M F32 weights and matches the CPU backend.

| Kernel | Commit | Notes |
|---|---|---|
| `matmul_f32` | `91ff03d` | 2D dispatch GEMM, 9 test shapes |
| `matmul_q8_0` (GEMV) | `97d7a7e` | Decode GEMV, full 49,152×576 lm_head = 892 ms |
| `rmsnorm_f32` | `a4c06ff` | Workgroup tree reduce |
| `rope_f32` + NeoX | `912e3d4`, `e807fd3` | Norm + NeoX (4 extra parity tests added in wave 3) |
| `attention_f32` | `268625e` | Flash-attention online softmax, TILE_KV=256 |
| `swiglu_f32` | `5ed52c3` | Trivial pointwise |
| `matmul_q8_0_gemm` | `d649171` | Batched Q8_0 matmul, ~209 GFLOPS at Llama-3 4096² N=64 |
| `rmsnorm_f32_sg`, `attention_f32_sg` | `19950b2` | Subgroup-arithmetic variants (probed + dispatched at runtime) |

All validated to abs 1e-4 / rel 1e-3 against CPU reference on AMD Radeon 8060S iGPU.

**Subgroup fast path finding (`81a23d0`):** numerically equivalent but *slower* on RDNA3.5 at WG=256 (attention 0.50×, rmsnorm 1.02× at SmolLM shapes) — barriers are cheap on modern AMD; extra register pressure + broadcast-slot barrier eats the log₂(256)=8 barriers saved. Kept in tree for future A/B on NVIDIA/mobile; default stays shared-mem. Toggle with `DOTLLM_VULKAN_FORCE_SHARED_REDUCE=1`.

**End-to-end Vulkan forward (`dc334d3`, `d7accc7`):** `VulkanTransformerModel` + `VulkanWeights` + `VulkanKvCache` + `VulkanForwardState`. Dequants Q8_0 to F32 at load (Q8_0 end-to-end is the next workstream). Rejects MLA/MoE/SSM at load time. 8/9 strict argmax matches CPU across prefill + 8 decode steps on `"The capital of France is"` (both backends → ` Paris`); the one "miss" is a top-2 rank swap from Q8_0-vs-FP32 precision noise, not a correctness bug (Vulkan path runs closer to an FP32 oracle).

**Descriptor-pool leak fix (`5490f34`):** all 7 wave-1/wave-2 kernels leaked descriptor sets — pool was `maxSets=1`, second `Launch` would die with `VK_ERROR_OUT_OF_POOL_MEMORY`. Per-kernel unit tests missed it (each test creates a fresh kernel); only the end-to-end integration surfaced it. Fix: `vkResetDescriptorPool` after `vkQueueWaitIdle` in every kernel's finally.

## Open items (check GitHub Issues — filed this session)

The open follow-ups have been filed as GitHub issues on `jamesburton/dotLLM` so they don't rot in this doc. Pull the issue tracker for the current state; the summary below is for orientation.

### Vulkan (GPU path — the biggest absolute remaining headroom)

- **Q8_0 end-to-end** — currently being worked on in parallel. Keep Q8_0 blocks on device, skip FP32 dequant at load; route matmuls through `matmul_q8_0` (GEMV) + `matmul_q8_0_gemm` (batched). Once this lands, Vulkan decode should beat CPU decode on this iGPU.
- **Perf wave** — currently being worked on in parallel. Device-local VRAM (staging ring), fence-based pipelining, descriptor pool reuse, remove `vkQueueWaitIdle` per kernel. Current Vulkan forward is correctness-first and runs in "seconds per forward" — not because the kernels are slow, but because every dispatch does a full host↔device round-trip.
- **Cooperative-matrix Q8_0 GEMM variant** — RDNA3 WMMA 16×16×16 int8 tiles via `VK_KHR_cooperative_matrix` should yield ~10× over current 209 GFLOPS on the batched kernel. Probe pattern from `e6f5e40` (subgroup probe) is the reference.
- **Arch ports**: MLA (DeepSeek-V2/V3), MoE (Mixtral, Qwen-MoE, DeepSeek-V2-MoE), SSM/Mamba-3/Nemotron-H hybrid. All rejected at load today.
- **Missing leaf kernels**: `per_head_rmsnorm_f32` for Qwen3 QK-norm; `bias_add_f32` for Phi-3-family biases (currently host-fallback).
- **Latent GEMV stride bug at K=32 with M>1** — `rowByteStride = rowUints * 4` is wrong when `blocksPerRow*34 % 4 != 0`. Not exercised by current tests; GEMM kernel works around it. File a test that exposes it + patch the GEMV.
- **Subgroup A/B across vendors** — on AMD RDNA3.5 the subgroup path is a loss; NVIDIA Ampere/Ada (subgroup=32) and mobile likely flip. `DOTLLM_VULKAN_FORCE_SHARED_REDUCE=1` lets per-vendor comparison happen without code changes.

### CPU (much smaller remaining headroom)

- **Prefill gap residual**: 2.4× to llama.cpp. Candidates: R4/interleaved path for N>1 prefill (currently decode-only); `GemmTiledF32Worker` / `GemmTiledF16Worker` need the same 2D-partition fix as Q8 got in `f9b1ff6`.
- **First-call-after-warmup noise** in `PrefillBenchmarks` — 3–5× on the first kernel call. Likely first-parallel-call scheduling / L1 cold on weight tiles. Visible as first-request latency at server start.

### Model/arch correctness (low priority, no current test failures)

- ROCm/HIP gfx1151: blocked on driver. Try WSL2 + `rocm/dev-ubuntu-24.04:6.4-complete`.
