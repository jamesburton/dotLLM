## The driver finding

The Strix Halo + Windows + amdvlk stack advertises `VK_EXT_external_memory_host` (extension enumerated, `minImportedHostPointerAlignment = 4096` reported via the properties chain) and accepts the extension at `vkCreateDevice` time. But it **rejects** `vkAllocateMemory` with `VK_ERROR_INVALID_EXTERNAL_HANDLE` (VkResult `-1000072003`) for every GGUF tensor pointer.

Reproduce:

```powershell
dotnet build benchmarks/DotLLM.Benchmarks/DotLLM.Benchmarks.csproj -c Release
dotnet run --project benchmarks/DotLLM.Benchmarks -c Release -- profile-vulkan-host-import
```

Output excerpt:

```
Device: AMD Radeon(TM) 8060S Graphics (vendor=0x1002)
HasExternalMemoryHost: True
minImportedHostPointerAlignment: 4096 bytes
─── Run B: host-import zero-copy ───
  [host-import] Fallback reason (last): import_rejected
  [host-import] Import failure detail: stage=vkAllocateMemory VkResult=-1000072003
  [host-import] Matrices: zero-copy=0  staging=155
```

The rejection is consistent across both `VK_EXTERNAL_MEMORY_HANDLE_TYPE_HOST_ALLOCATION_BIT_EXT` and `VK_EXTERNAL_MEMORY_HANDLE_TYPE_HOST_MAPPED_FOREIGN_MEMORY_BIT_EXT` — `HostVisibleBuffer.TryCreate` iterates both. The framework's fallback to staging fires transparently on every matrix; weights still load correctly, just no faster.

## Evidence the framework is correct

`tests/DotLLM.Tests.Unit/Vulkan/VulkanHostImportParityTests.cs` allocates a page-aligned host buffer with `NativeMemory.AlignedAlloc`, quantises random F32 weights into it as Q8_0 blocks, then runs the same matmul kernel against:

1. The bytes uploaded via the staging path.
2. The same bytes imported zero-copy via `HostVisibleBuffer`.

Both runs produce **bit-identical output** (compared via `BitConverter.SingleToInt32Bits`). On Strix Halo this passes — the import succeeds for heap-allocated memory, only the mmap'd file pointer is rejected.

Tests pass on Strix Halo:

```
Passed!  - Failed:     0, Passed:     8, Skipped:     1, Total:     9
```

The 1 skip is `TryCreate_ReturnsNull_WhenExtensionAbsent` — moot when the driver supports the extension.

The full Vulkan test suite (`Category=GPU&FullyQualifiedName~Vulkan`) also runs clean: **375 pass, 2 skip, 0 fail**. No regressions from the host-import wiring.

## Why amdvlk rejects mmap'd file pages

Unconfirmed but consistent with the `VK_EXT_external_memory_host` spec text requiring imported memory be "readable and writable from the host". The dotLLM GGUF loader opens the file with `MemoryMappedFileAccess.Read` — read-only OS-level mapping. Heap-allocated pages from `NativeMemory.AlignedAlloc` are read-write and accepted.

Changing `GgufFile.Open` to use `MemoryMappedFileAccess.ReadWrite` or `CopyOnWrite` violates the task anti-goal "Don't change GGUF mmap semantics". A cleaner fix is a Vulkan-side secondary mmap (CopyOnWrite) but that doubles the page-table backing — not free.

## What to try next

1. **Mesa radv on Linux Strix Halo.** Mesa accepts foreign-memory imports of mmap'd ranges since 18.3. If the import succeeds there, the framework is the win on Linux out of the box. Test on the same hardware booted to Ubuntu 24.04+ Mesa 24+.
2. **CopyOnWrite Vulkan-only mmap.** Open a SECONDARY `MemoryMappedFile.CreateFromFile` with `MemoryMappedFileAccess.CopyOnWrite` just for the Vulkan import. The CPU mmap stays read-only. Page tables are duplicated but physical pages still share until a write — and no writes happen on weight pages. Costs a few hundred KiB of page-table overhead, gains zero-copy on amdvlk.
3. **amdvlk driver report.** File against `https://github.com/GPUOpen-Drivers/AMDVLK` describing the rejection of read-only mmap pages with `HOST_MAPPED_FOREIGN_MEMORY_BIT_EXT`. The spec says foreign-memory means "non-Vulkan-allocated memory mapped into the process address space" — read-only mmap'd files literally are that.
4. **NVIDIA discrete on PCIe BAR.** On NVIDIA the import maps the host page into the BAR aperture. Worth measuring whether it's a win on a 4090 (probably not vs PCIe DMA) and on a Jetson (unified memory APU, like Strix Halo) — likely a big win.
5. **Wire MemoryDomain through `IBackend`.** The `MemoryDomain` enum landed but is only consumed by the Vulkan path. Threading it through `IBackend.AllocateTensor` / `LoadTensor` would let CPU and CUDA backends also report their effective domain — useful for diagnostics and for future placement decisions.
6. **Apply to MoE expert banks.** The packed-bank layout currently does host-side packing of per-expert F32 matrices. On a UMA APU, packing into mmap'd pages first (writing the bank into a private mmap), then importing the whole bank, would still avoid the staging copy. Out of scope for this branch.

## Process RSS verdict (TinyLlama Q8_0, 1.12 GiB)

Despite the import being rejected on Strix Halo, the RSS measurements show the framework would behave correctly when import succeeds:

| Run | Wall ms | RSS delta | Zero-copy matrices |
|---|---|---|---|
| Staging (forced) | 733 | 1368 MiB | 0/155 |
| Host-import attempt | 765 | 1370 MiB | 0/155 (all rejected, fell to staging) |

Both runs counted the 1298 MiB of weights through staging on this driver. If a single matrix had imported successfully, its bytes would have been excluded from the RSS delta. Validating the RSS-saved acceptance criterion requires a driver that accepts the import — see "What to try next" item 1 or 2.
