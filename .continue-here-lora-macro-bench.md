# LoRA macro-bench fixtures

`benchmarks/DotLLM.Benchmarks/Lora/LoraMacroBenchmarks` resolves a real GGUF
checkpoint at runtime in priority order:

1. `DOTLLM_BENCH_MODEL_PATH` env var (manual override — preferred for
   per-machine repro).
2. TinyLlama GGUF anywhere under `~/.dotllm/test-cache/` (preferred fixture
   per the Phase 4d.3 spec — see commit `5dcd4ba` context).
3. `~/.dotllm/test-cache/bartowski/Llama-3.2-1B-Instruct-GGUF/Llama-3.2-1B-Instruct-Q8_0.gguf`
   (closest available stand-in — same architecture family, similar scale).
4. `~/.dotllm/test-cache/QuantFactory/SmolLM-135M-GGUF/SmolLM-135M.Q8_0.gguf`
   (smallest fallback).

If none of the above exists locally, the bench skips cleanly — the
`GlobalSetup` prints a `[LoraMacroBenchmarks] SKIP` line and every
`Benchmark` returns an empty `InferenceResponse` immediately. The BDN
suite stays green; the custom tok/s columns show "N/A".

## To run against TinyLlama-1.1B specifically

Download once into the test-cache, then re-run:

```pwsh
# Either point at the existing safetensors checkpoint at C:/temp/dotllm-tinyllama
# (won't be picked up automatically — needs DOTLLM_BENCH_MODEL_PATH override)
# OR download a TinyLlama GGUF (e.g. TheBloke/TinyLlama-1.1B-Chat-v1.0-GGUF
# Q8_0 ~1.1 GB) into ~/.dotllm/test-cache/.../ for auto-discovery.

$env:DOTLLM_BENCH_MODEL_PATH = "C:\path\to\tinyllama-1.1b.Q8_0.gguf"
dotnet run -c Release --project benchmarks/DotLLM.Benchmarks `
    -- --filter '*LoraMacroBenchmarks*' --invocationCount 1 --unrollFactor 1
```

The synthetic adapter geometry adapts to the loaded base's `ModelConfig`
(layer count + per-projection dims) — no checkpoint-specific code changes
needed.

## Anti-goals

- The bench does NOT trigger a download. If `DOTLLM_BENCH_MODEL_PATH` is
  unset AND no cached GGUF matches, the bench skips. This keeps the
  measurement-only follow-up free of network and disk-cap concerns.
- The bench does NOT ship a real LoRA adapter file. The synthetic adapter
  in `SyntheticLoraAdapter` exercises the same dispatch path with
  deterministic noise; using a real adapter wouldn't change the perf
  delta (which only depends on shape).

## Results captured

See `docs/LORA.md` → "Performance — Macro-Bench (Phase 4d.3)" section.
Run was on Llama-3.2-1B-Instruct.Q8_0 (TinyLlama-1.1B GGUF not present
in this machine's test-cache). The +9% kernel-level prefill regression
amplifies to ~36% at the system level when LoRA is applied on top of a
quantised (Q8_0) base — the F32 LoRA delta has a worse FLOP/byte ratio
than the quantised base GEMM.
