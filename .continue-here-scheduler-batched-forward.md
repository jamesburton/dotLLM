# Step 35 follow-up: backend-side ForwardBatch overrides

## What landed in this round (commits `d9190d5` … `736a217`)

- `IModel.ForwardBatch(IReadOnlyList<SequenceForwardRequest>, deviceId)` —
  API signature with a default interface method that loops over the
  per-sequence `Forward`. **No backend overrides yet.**
- `SequenceForwardRequest` — per-seq tokens / positions / KV-cache / adapter.
- `ContinuousBatchScheduler.Step()` now bundles ≥2 active sequences into a
  single `ForwardBatch` call and processes the returned tensors in order.
  Single-sequence iterations still go direct through `Forward` to avoid the
  batch-allocation overhead and preserve `TextGenerator`-equivalent latency
  for single-tenant decode.
- Chunked prefill via `MaxPrefillTokensPerStep`. Tested.
- Server endpoints + scheduler `RunLoopAsync` background task + engine
  telemetry providers all wired. Tested.

## What's left

The current default `ForwardBatch` is correct but slow — it pays N kernel
dispatches per Step when N sequences are active, when one fused dispatch
would suffice. The throughput win of continuous batching is mostly
locked behind the backend overrides below.

### CPU (`TransformerModel.ForwardBatch`)

- Each sequence's attention is independent (own K/V). Attention stays per-seq.
- The matmuls (Q/K/V/O projections, gate/up/down FFN, embedding, lm_head)
  *can* be batched: pack all per-seq token counts into one big GEMM with
  shape `[Σ N_i, hidden_size] × [hidden_size, dim_out]`. Then split the
  output back into per-seq tensors before the attention call.
- `TensorPrimitives.MatrixMultiply` doesn't have a batched variant —
  but a single large GEMM is faster than N small GEMMs on multi-core CPUs
  because of better cache reuse and OpenMP-style parallelism via
  `Parallel.For` over the output rows.
- LoRA: each adapter call already factors per-token; if all sequences
  share the same adapter the LoRA matmul also batches, otherwise per-seq
  fallback within the batched layer.

### Vulkan (`VulkanTransformerModel.ForwardBatch`)

- Existing kernels already handle `[N, hidden]` inputs — extension is to
  let `N` span sequences. Need a per-seq position-offset uniform block so
  RoPE picks the right rotation per position.
- Attention is the harder part: separate K/V buffers per sequence. Two
  options:
  1. **Per-seq attention loop** (cheap): keep the existing attention
     kernel, dispatch it once per sequence. Wastes the per-sequence
     dispatch but matmuls still batch.
  2. **Per-seq mask / block-table attention** (vLLM-style): one attention
     kernel reads block tables to find each sequence's K/V. Bigger lift,
     bigger win.

### CUDA (`CudaTransformerModel.ForwardBatch` / `HybridTransformerModel`)

- Same pattern as Vulkan. Path 1 (per-seq attention loop) likely lands
  first because the existing cuBLAS GEMM call already accepts arbitrary M,
  and attention kernels just need to be dispatched in a stream-per-seq
  fashion.

## Acceptance / verification

- The scheduler-side dispatch contract is already tested
  (`FourConcurrentSchedulerTests`). The same test exercises the default
  per-sequence-loop path today — when the backend override lands it must
  produce **byte-identical logits** to the current default for any
  fixed batch.
- A microbench comparing decode tokens/sec with `MaxActiveSequences=1`
  vs `MaxActiveSequences=8` will quantify the win. Today both are equal
  (per-seq loop); after the override, expect 4-7x in the decode batch.
