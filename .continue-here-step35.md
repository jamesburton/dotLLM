## What is NOT done

These are explicitly deferred — none are in Step 35 scope per the roadmap, but the next agent
should know:

1. **Kernel-level batched forward across sequences.** `IModel.Forward` is still called once per
   sequence per iteration. The scheduler buys you KV-pool sharing, admission overlap, and the
   ability to start a new sequence the moment one finishes — but not yet the GEMV-batched decode.
   Adding the latter requires the model interface to accept per-token routing into per-sequence
   KV-caches, which is non-trivial. **Step 59** (advanced scheduling) is the right place.

2. **Chunked prefill.** A long prompt is prefilled in a single forward pass. The
   `MaxPrefillTokensPerStep` option only bounds *which sequences are admitted per step*, not
   *how much of one prompt is prefilled per step*. Adding chunked prefill requires the model to
   accept partial prefills with an explicit cache_position — which the API already supports —
   plus a per-sequence "prefill cursor" in `SchedulerRequest`. Skeleton:
   ```csharp
   // in SchedulerRequest:
   int PrefillCursor;  // number of prompt tokens already prefilled
   // in ContinuousBatchScheduler.AdmitAndPrefill:
   int chunk = Math.Min(remaining, options.PrefillChunkSize);
   // run Forward over promptIds[cursor..cursor+chunk)
   // keep state == Prefilling until cursor == promptLen, then sample + transition to Decoding
   ```

3. **Streaming yield.** Generated tokens accumulate inside `SchedulerRequest.GeneratedTokens`
   and surface only when `Completion` resolves. SSE-style streaming through the scheduler needs
   a per-request `ChannelWriter<GenerationToken>` populated inside `DecodeOneStep` after the
   stop check. Drop-in change.

4. **Stop-string conditions.** EOS + MaxTokens both work. `StopStringCondition` is accepted
   but never fires — the MVP doesn't carry a per-sequence `IncrementalDetokenizer`, so the
   decoded-tail span passed to `IStopCondition.ShouldStop` is empty. Adding it is mechanical
   (allocate one detok per `SchedulerRequest`, mirror the stop-tail computation in
   `TextGenerator.ComputeStopTailSize`).

5. **Preemption / swap.** `SchedulerMetrics.PreemptionCount` is wired but never incremented;
   sequences run to completion once admitted. Step 59 owns this.

6. **Logprobs.** `InferenceOptionsLike` carries `Logprobs` / `TopLogprobs` flags but they're
   not consumed yet — the scheduler ignores them. Need to thread `LogprobsCapture` through
   `DecodeOneStep` and stash per-token info on `SchedulerRequest` in the same shape
   `TextGenerator` produces.

7. **Server wire-up.** `docs/SERVER.md` documents a `SemaphoreSlim(1,1)` request gate. The
   scheduler is ready behind `ContinuousBatchSchedulerService`; the endpoint just needs to
   stop holding the gate and instead call `EnqueueAsync`. A small `IServiceCollection`
   helper that registers the singleton service + a `IHostedService` driving `RunLoopAsync`
   would land that cleanly. Keep `TextGenerator` reachable as the single-request path for
   the `chat` CLI subcommand — it's intentionally not on the scheduler path because the
   prompt-cache integration there isn't equivalent yet.

8. **Prefix cache integration.** `TextGenerator` reuses `SimpleKvCache`/`PagedKvCache`
   instances through `PrefixCache.FindMatch`. The scheduler allocates a fresh cache per
   admission and ignores the prefix cache entirely. Step 37 (advanced prompt caching)
   will replace per-sequence allocation with prefix-trie-based block sharing — the
   per-sequence KV-cache pointer is intentionally not finalized until then.

## Design decisions locked in

- **Two-layer scheduler interface.** `IBatchScheduler` is the synchronous step-driven API
  used by tests and by the inner loop. `IScheduler` (existing, async) is implemented by
  `ContinuousBatchSchedulerService` which wraps the sync API. Keeps the hot path tight and
  doesn't force `async`/`await` on each iteration.
- **Per-sequence sampler pipeline.** RNG state is per-sequence — sharing one pipeline across
  sequences would tangle their token streams under non-greedy sampling.
- **KV-cache via factory.** The scheduler doesn't know whether the cache is paged or simple.
  `Func<ModelConfig, int, IKvCache>` returns a fresh per-sequence cache; the caller wires
  `PagedKvCacheFactory.Create` (preferred for serve) or the default `SimpleKvCache`.
- **Forward stays per-sequence in MVP.** Batched forward would change the `IModel` contract;
  out of scope. The scheduler benefits from admission overlap and pool sharing alone.
- **Cancellation.** Caller-side `CancellationToken` flips state to `Cancelled` and trips
  the `CompletionSource`. `Step()` later sweeps cancelled sequences and releases blocks —
  it would be unsafe to free the KV-cache inside the cancellation callback because the
  scheduler thread may be mid-`Forward`.

## How to extend (likely Step-59 follow-up)

To add kernel-batched decode without disrupting the MVP shape:

1. Introduce `IBatchedModel : IModel` with `Forward(BatchView batches, ...)` where `BatchView`
   is a flat token+position pair with a per-token sequence index. Each backend opts in.
2. In `ContinuousBatchScheduler.Step()`, before the per-sequence decode loop, group active
   sequences into a single `BatchView`, dispatch one `Forward`, then split logits back to
   per-sequence sampling. KV-cache routing inside the kernel becomes "which block to update
   for token i" — straightforward with `PagedKvCache`.
3. The per-sequence path stays as the fallback for non-batched backends and for prefill
   (which is already one big forward today).
