# BLOCKING CONSTRAINTS — Read Before Anything Else

> These are not suggestions. Each constraint below was discovered through failure this session.
> Acknowledge each one explicitly before proceeding with related work.

- [ ] CONSTRAINT: **gguf-py is NOT an independent correctness oracle for dotLLM** — both are independent reimplementations of llama.cpp's ggml semantics, but they CAN (and did) share the same convention bug. tools/parity_check*.py says "PASS" while the model produces garbage. Authoritative oracle is llama.cpp's actual CPU kernels (`ggml/src/ggml-cpu/ops.cpp`) and `llama-eval-callback` per-tensor dumps. — Structural mitigation: cross-backend semantic verification PRs must include per-tensor parity against llama-eval-callback via `tools/bin_diff.py`, not just `tools/parity_check*.py`.

- [ ] CONSTRAINT: **Test fixtures must use shapes that discriminate the parameter in question.** All pre-fix GDN unit tests used NVHead=2, NKHead=1 where `vh/vHeadsPerKHead` and `vh%NKHead` both equal 0 — neither could detect a tiled-vs-interleaved head-broadcast bug. CI ran green while the production model produced gibberish. — Structural mitigation: when a kernel takes parameters that can degenerate (e.g. NKHead=1 collapses head broadcasting), at least one fixture must use values where the discrimination matters. `GatedDeltaNetScanTests.HeadBroadcast_IsTiledNotInterleaved_QwenMoeStyle` (added this session, fdb39b4) is the template — NVHead != NKHead AND NVHead/NKHead > 1 AND per-kh inputs distinct.

**Do not proceed until both boxes are checked.**

## Critical Anti-Patterns

| Pattern | Description | Severity | Prevention Mechanism |
|---------|-------------|----------|---------------------|
| Trusting gguf-py parity as an independent oracle | parity_check.py / parity_check2.py / layer_moe_sweep.py / etc. ALL agreed dotLLM was correct because they shared the same head-broadcast convention bug. Spent ~2 hours convinced dotLLM was correct before realising both gguf-py and dotLLM are independent reimplementations of the same upstream semantics — neither is "ground truth". | blocking | Use llama-eval-callback per-tensor dumps as ground truth via tools/bin_diff.py for cross-backend correctness PRs. tools/bin_diff.py's element-wise SNR/cos comparison is the discriminator. |
| Test fixtures that degenerate the parameter under test | NVHead=2, NKHead=1 was used in 6 of 7 GDN tests. With NKHead=1 the head-broadcast mapping always returns kh=0 regardless of formula — the bug was untestable. | blocking | When a kernel parameter has degenerate values (NKHead=1, seqLen=1, batch=1, dim=1), at least one fixture must use a non-degenerate value AND inputs distinct enough that wrong-mapping produces a numerically different result. |
| Cross-backend duplication fixed in only one backend | The GDN scan bug was identical in CPU (.cs), CUDA (.cu + .ptx), and Vulkan (.comp). Fixing only one backend leaves a time bomb — diagnostic info evaporates between sessions. | advisory | CLAUDE.md "Cross-Backend Critical Bugs" section codifies the rule: search across src/DotLLM.{Cpu,Cuda,Vulkan,Hip}/, native/kernels/, native/vulkan/shaders/ before declaring done. |
| Believing the task framing without falsifying it | Task said: "CRITICAL correctness regression at commit 71eaae4 in MoE perf refactor". Bit-exact dump compare of HEAD vs pre-71eaae4 showed 71eaae4 was a no-op for output. The regression is much older. Falsifying premises early would have saved hours of MoE code review. | advisory | When framed a regression task, run bit-compare against the pre-regression reference BEFORE diving into the suspect commit. Time spent confirming the regression is real and where it is, is much cheaper than chasing it in the wrong code. |
