# Continue-here: Vulkan-on-Strix-Halo perf baseline

This worktree (`worktree-agent-a480e8ccd7f9935e4`) was tasked with
characterising Vulkan inference performance on Strix Halo and identifying
the top RDNA3.5-specific optimisation opportunities.

## What landed

1. **Bench harness**: `benchmarks/DotLLM.Benchmarks/InferenceBenchmarks.cs`
   gained a `vulkan` device path mirroring the existing `gpu` (CUDA) path
   (`LoadVulkanModel`, `[NoInlining]` so the Vulkan assembly isn't loaded
   on the CPU-only path). `scripts/bench_compare.py` `--device` accepts
   `vulkan`. `DotLLM.Vulkan` is now a project reference of `DotLLM.Benchmarks`.

2. **Vulkan baseline JSONs** (under `docs/perf/`):
   - SmolLM-135M Q8_0
   - Llama-3.2-1B-Instruct Q8_0 (the headline measurement, CV 1.2%)
   - Bielik-1.5B-v3.0-Instruct Q8_0 (Llama-arch stand-in for the 3B slot)
   - Qwen2.5-0.5B-Instruct Q8_0 (Qwen-family stand-in for Qwen3.6-A3B)
   - Same-machine CPU baselines for SmolLM-135M and Llama-3.2-1B Q8_0
     (so the "Strix Halo Vulkan vs Strix Halo CPU" ratio is on one host).

3. **`docs/PERFORMANCE.md` §6** — Vulkan baseline + cross-machine reading
   + top-3 headroom items + reproduction commands.

4. **`benchmarks/VulkanGemvWorkgroupSweep`** — the workgroup-size
   sensitivity microbench for Q8_0 GEMV. Three SPV variants were added
   under `native/vulkan/spv/matmul_q8_0_{wg64,wg256,sg}.spv` (sources
   under `native/vulkan/shaders/`). The bench rotates them in/out of the
   production `matmul_q8_0.spv` filename so it doesn't need to fork the
   kernel class. Negative result: WG=128 + shared-mem reduce (current
   production) is already at the local optimum on Strix Halo.

## What didn't land (blockers, scope cuts)

- **Qwen3.6-A3B Q8_0 baseline**. The model isn't cached on this host
  (`~/.dotllm/test-cache/Qwen/` has only `Qwen2.5-0.5B`). Per the brief's
  "don't trigger downloads" policy, used Qwen2.5-0.5B Q8_0 as a Qwen-
  family stand-in. To complete the brief, drop a copy of any
  `Qwen3.6-35B-A3B-UD-*.gguf` (Q8_0 ideally; Q6_K_XL or Q4_K_M acceptable)
  into `~/.dotllm/test-cache/` and re-run:
  ```
  python scripts/bench_compare.py \
      --model <path-or-repo-id> --quant Q8_0 \
      --device vulkan --tokens 30 --iterations 3 --runs 3 --prompt-size short \
      --label "vulkan-strixhalo-qwen36-a3b-q8" \
      --dotllm \
      --export-json docs/perf/baseline-qwen36-a3b-vulkan-strixhalo.json
  ```
  Note the GDN multi-token shader needed the `665f228` "rename `active`
  → `laneActive`" fix to compile on this `glslc`/`shaderc` (newer
  reserved-words list). That backport is in this worktree's tree.

- **llama.cpp Vulkan-on-Strix-Halo side-by-side**. No llama.cpp Vulkan
  build was available on this host. Build llama.cpp with
  `-DGGML_VULKAN=ON` against the same Vulkan SDK and re-run the
  bench_compare commands in PERFORMANCE.md §6.5 with `--llamacpp` and
  `--llamacpp-bin <path>`.

- **Llama-3.2-3B Q8_0 baseline**. Not cached on this host (model lives
  on the prior baseline machine only). Bielik-1.5B was used as a
  stand-in for the larger-Llama slot.
