# Quantised LoRA delta — design notes & spike

## Background

Agent 8's macro-bench (commit `9c74060` on `worktree-agent-a1ceece8903b019ca`)
showed that on a Q8_0 base, an F32 LoRA delta drops Strix Halo prefill
throughput by ~36% (vs ~9% predicted by the kernel-level bench). Cause: the
F32 LoRA delta has a much worse FLOP/byte ratio than the Q8_0 base GEMM —
the base is now memory-bandwidth-bound and very fast per FLOP, so the F32
delta becomes a large relative share of the forward pass.

## Approaches considered

**A. Quantise LoRA weights to Q8_0** at adapter load time. Runtime
dequantises on the GEMV inner loop, mirroring the base-model Q8_0 GEMV.

**B. Fused dequant-GEMM** that consumes the F32 LoRA but pipelines the
dequant of the base activation through the LoRA delta in one pass.

## Decision: Approach A (with B-only Q8_0)

### Why A

- Directly attacks Agent 8's diagnosed root cause (LoRA byte volume too
  high for the contracted-FLOP count).
- Reuses dotLLM's proven `GemvQ8_0` / `GemmQ8_0` / `QuantizeF32ToQ8_0`
  kernels with their existing AVX-512 / AVX2 paths.
- Mirrors the F16 / BF16 path landed at `9864bc6` (Phase 4d.1) — the
  enum-driven dispatch in `LoraDelta.Apply` already supports asymmetric
  per-buffer dtypes.
- Bit-parity tolerance is well understood (~5e-3 abs vs F32 ref) — same
  as F16, no new tolerance regime needed.

### Why **not** B

- B reduces base-weight re-loads, but base weights are already loaded
  once per projection per forward — the redundant-load cost is small.
- B doesn't reduce the LoRA byte volume itself (which IS the bottleneck
  per Agent 8's diagnosis).
- B requires invasive changes to the per-projection accumulator path,
  breaking the kernel/strategy boundary.

### Why B-only Q8_0 (not also A)

- B has shape `[rank, inputDim]` — `inputDim` is `hidden`, `qOut`, `kvOut`,
  `ffn` etc., always a multiple of 32, so each row is one natural Q8_0
  vector.
- A has shape `[outputDim, rank]` where the contracted axis is **rank**
  (typical 8/16/32/64). Q8_0 requires 32-element blocks; rank<32 is the
  common case. Padding A rows to 32 with zeros wastes 50%+ storage at
  rank=16 — the savings vs F16 (1.0625B/elem padded vs 2.0B/elem) collapses.
- B and A are similar size in bytes, **but B is the bandwidth bottleneck**:
  for stage 1, each `x[t,:]` is reused only `rank` times across B rows
  (16x for typical rank=16); for stage 2, each `tmp[t,:]` is reused
  `outputDim` times across A rows (2048x for typical hidden=2048). A
  enjoys 128x more reuse than B per row, so it spills to L1 sooner.
- Conclusion: quantise B to Q8_0, leave A as F16 (or whatever the loader
  shipped). Aggregate byte volume drops from `2 × 4 × hidden × rank`
  (F32 both) to `(1.0625 + 2.0) × hidden × rank` — ~2.2× total LoRA byte
  reduction. Adding A→Q8_0 with padding would push this to ~2.6× for
  ~5% extra plumbing — deferred to a follow-up if numbers warrant.

## Implementation plan

1. Extend `LoraWeightDType` with `Q8_0 = 3` (foundational).
2. Add `Quantize_F32_To_Q8_0` adapter-load helper (one-shot, one kernel
   call per projection — no need for hot-path tuning).
3. Extend `LoraDelta.Apply(void*, void*, ...)` to dispatch on the new
   dtype: when bDType=Q8_0, use `GemmQ8_0` for stage 1 (input quantises
   x once per call, reuses across `rank` rows). Stage 2 unchanged.
4. Extend `PeftAdapterLoader` with a `quantizeBToQ8_0` opt-in. Wire
   through to `SyntheticLoraAdapter` so the bench can build Q8_0 adapters
   without a real PEFT checkpoint.
5. Add `LoraQ8_0Variant` to `LoraVariant` enum in `LoraMacroBenchmarks`
   so the bench measures the new path under the same harness.
6. Bit-parity unit test (`LoraDeltaQuantizedQ8_0Tests`) — Q8_0 dispatch
   matches F32 reference within Q8_0 tolerance (~1e-2 abs / 1e-2 rel for
   small-scale uniform random adapters).

## Hardware target

Strix Halo (Ryzen AI Max+ 395, 16C/32T, AVX-512F+CD+BW+DQ+VL+VBMI). Same
machine + same `LoraMacroBenchmarks` harness as Agent 8, so deltas are
directly comparable.

## Acceptance gate

- Unit test `LoraDeltaQuantizedQ8_0Tests.Q8_0_MatchesF32Reference` passes
  within stated tolerance.
- Existing `LoraDeltaQuantizedDtypeTests` + `LoraForwardParityTests` +
  `PeftAdapterLoaderTests` still green.
- Macro-bench rerun: Q8_0 LoRA prefill regression ≤ −10% vs NoLora
  (Agent 8 measured −36% for F32 LoRA on the same config).

## Anti-goals

- No Vulkan-side change in this agent. Vulkan iGPU exercise of the new
  path is a follow-up agent (per spec).
- No LoRA dispatch / scheduler / batcher seam changes.
- No premature abstraction — the dtype enum + a third dispatch arm in
  `LoraDelta.Apply` is the entire surface area of this change.
