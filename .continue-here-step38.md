## Open items — please wire before exposing to real callers

### 1. API-key authentication is a stub

`HeaderApiKeyResolver` reads `X-API-Key` first, then
`Authorization: Bearer <token>`, and returns `"anonymous"` for
unauthenticated traffic. **This is not authentication** — it only exists so
rate-limit buckets can be partitioned per caller. Any caller can claim any
key.

When you wire real auth (OAuth/JWT/mTLS — out of scope here per Step 38's
anti-goals), register your own `IApiKeyResolver` implementation that
returns the authenticated principal's stable ID:

```csharp
builder.Services.AddSingleton<IApiKeyResolver, MyJwtSubResolver>();
```

The resolver runs *after* whatever auth middleware you install. The
rate-limit machinery is transport-agnostic and will bucket on whatever
opaque string you return.

### 2. Token-budget refund is best-effort

`RateLimitManager.TrueUpTokens` records the unused-token delta but
**cannot currently push tokens back into the `TokenBucketRateLimiter`** —
the BCL doesn't expose a public refund API for partition-managed
limiters. The reservation acts as a cap, which is the correct semantic
for billing safety. If/when .NET adds a `Replenish(int amount)` overload
we can wire it in; the `RecordRefund` hook is already in place.

### 3. Token estimation is heuristic

The middleware doesn't run the tokenizer at admission time — it uses
`max(prompt_chars / 4, 1) + max_tokens` (or
`EstimatedCompletionTokensFallback` when `max_tokens` is unspecified).
This is conservative and gets trued-up against the authoritative count
after generation. If exact accounting is required at admission, tokenize
the body in the middleware (cost: ~hundreds of µs per request for typical
BPE tokenizers).

### 4. Cross-request priority preemption is NOT here

`RequestPriority` only affects admission queueing under the concurrency
cap. The middleware does **not** interrupt in-flight generations to make
room for higher-priority requests — that's a scheduler concern and lands
in Step 59 (Advanced scheduling). Today, with the single-request gate, a
`Critical` request still has to wait for the current generation to
complete; it just jumps the queue.

### 5. No CLI surface yet

`ServerOptions.RateLimit` is currently only loadable
programmatically — `ServeCommand` doesn't expose CLI flags for the
config. Add `--rate-limit-config <path.json>` (or wire it through
`builder.Configuration.GetSection("RateLimit")`) when the CLI gets a
configuration story.

### 6. Metrics integration

When Step 45 lands, the rate-limit manager should emit
`System.Diagnostics.Metrics` counters for admissions / rejections per
limiter kind per API key. The plumbing point is `RateLimitManager.TryAcquireAsync`
return path and `KeyState.RecordRefund`.
