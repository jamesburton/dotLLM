## Summary

The task as briefed cannot be implemented as a single shader + wrapper because
its premise does not hold for the Vulkan backend in this tree. The handoff
note (`.continue-here.md` § "LoRA follow-ups") states:

> **Fused `RmsNormQuantizeAndKeepF32` shader** to recover the LoRA-active
> fused-rmsnorm bypass cost in the Vulkan decode path. Not blocking but caps
> perf headroom.

Two assumptions inside that one sentence are wrong against the actual code:

1. **Vulkan has no input-quantization step.** The CPU/CUDA forward path has
   a `FusedOps.RmsNormQuantize{Q8_0,Q8_1,Q8_K}` family
   (`src/DotLLM.Cpu/Kernels/FusedOps.cs:111+`) that emits a *quantized* (Q8_0
   / Q8_1 / Q8_K) input scratch into which the subsequent matmul consumes
   pre-quantized activations. Vulkan matmul shaders (`matmul_q8_0.comp`,
   `matmul_q4_k_gemv_f32.comp`, etc.) all take **F32 activations** and do the
   activation × dequantized-weight dot product in floats. There is no
   "RmsNorm → Quantize" pass anywhere in Vulkan to bypass.

2. **No LoRA bypass of the fused rmsnorm exists in Vulkan today.** The
   existing fused shader `native/vulkan/shaders/rmsnorm_matmul_q8_0.comp`
   (commit `3f45231`) already writes BOTH the F32 normalized hidden state
   (`normOutput`, used by K/V/Up and by LoRA delta) AND the matmul output
   (`y`). The dispatch site (`TryRecordFusedRmsNormMatmul`,
   `VulkanTransformerModel.cs:1064`) is unconditional on LoRA state — its
   gating predicates are only `seqLen == 1 && weightQt == Q8_0 &&
   inputDim ≤ MaxHiddenSize (1024)`. The LoRA delta sites at lines
   790, 849, 909, 935 explicitly read from `_state.NormOutput` (the F32 norm
   output the fused shader writes), with the in-source comment at lines
   786-789 stating:

   > // The fused rmsnorm+matmul path above still writes F32 normOut (its
   > // shader's contract), so the LoRA matmul's input is materialised in
   > // either branch — no need to bypass the fused path.

   That is, the LoRA-active and LoRA-inactive Vulkan decode paths take the
   same fused shader. There is no perf delta between them attributable to
   rmsnorm fusion.

## Where the handoff-note framing comes from (best guess)

The "Vulkan LoRA fused-rmsnorm bypass" item appears to be a mirror of the
CPU follow-up: in `src/DotLLM.Models/Architectures/TransformerModel.cs:577-590`
the CPU forward force-disables the `FusedOps.RmsNormQuantize` decode path
when an adapter is active, because that fused op overwrites the F32 norm
scratch with quantized bytes. The CPU follow-up wants a CPU
`RmsNormQuantizeAndKeepF32` op that produces both outputs. That problem is
real on CPU/CUDA but does not transfer to Vulkan, where the fused shader
already produces both outputs (the Vulkan side already solved the
"keep F32" requirement when `3f45231` landed, because `normOutput` was
already needed for K/V/Up — LoRA was a free rider).

## What an actually-meaningful Vulkan follow-up would look like

The genuine perf headroom on the Vulkan LoRA-active decode path is in the
**LoRA delta application** itself, not in the rmsnorm fusion. Each LoRA
delta site (`MaybeApplyLoraDelta`, `VulkanTransformerModel.cs:1671+`) is
**four** dispatches today (B-matmul → A-matmul → AddKernel → vkCmdCopyBuffer
to land in y) per adapted projection, per layer. For TinyLlama with all
7 projections adapted across 22 layers that is `7 × 22 × 4 = 616` extra
dispatches per decode token over the no-LoRA path. Fusing the
`Add + CopyBuffer` tail (a "scaled add into y" shader) cuts this to 3, and
fusing the B-matmul + A-matmul (a "rank-r low-rank GEMV" shader) cuts it
further to 1 — a much larger win than anything in the rmsnorm fusion. The
existing `MoeIndexedLoraDeltaF32Kernel`
(`src/DotLLM.Vulkan/Kernels/MoeIndexedLoraDeltaF32Kernel.cs`) is the
proof-of-concept shape for the MoE side; the dense side wants the same
treatment.

If "recover LoRA-active perf headroom in Vulkan decode" is the actual goal,
the work item should be retitled and rescoped to one of:

- **`LoraDeltaInlineAdd` fused shader** — `add + copy` collapsed into a
  single dispatch that writes back into `y` directly. Smallest win (~30%
  fewer dispatches per delta), smallest scope, no math changes.
- **`LoraDeltaGemvFused` shader** — one shader that does
  `y[m] += scale × sum_r A[m,r] · sum_k B[r,k] · x[k]` with `r ≤ 32` (the
  typical LoRA rank) in registers. Single dispatch per LoRA delta site.
  Replaces all four current dispatches. Largest win, largest scope.
- **`LoraDeltaGemvFused` for ranks ≤ 16 + small fallback** — the common
  PEFT defaults are rank=8 and rank=16; specialise for those and route the
  long-tail rank=32+ adapters through the existing path. Best
  effort/win ratio.

Any of these requires designing the kernel shape with the existing
`VulkanForwardState.LoraTmp / LoraDelta / LoraDeltaSum` scratch slots in
mind (and probably reducing them to just one once the chain collapses).

## Recommendation

Mark the original `RmsNormQuantizeAndKeepF32` Vulkan follow-up as **not
applicable** to this tree and remove it from the LoRA follow-up bundle.
Replace it with the `LoraDelta*` fusion follow-up above if the perf
headroom is genuinely needed. Otherwise this is a deliberate no-op.

The macro-bench follow-up (4d.3) and the MLA / MoE delta wiring follow-up
(4d.2) are unaffected.
